# Cube Worlds
Telegram Mini App game on TON blockchain. CUBE points via daily claims and referrals → NFT minting (admin-curated queue). Grammy bot + Fastify API + Vue 3 frontend + MongoDB/Typegoose.

## Commands
```bash
npm install && npm --prefix src/frontend install
npm run dev                                      # Backend watch (tsx)
npm --prefix src/frontend run dev                # Frontend dev (port 5173)
npm run build:all                                # Backend (tsc) + frontend (vite)
npm run lint && npm run typecheck && npm run test:backend  # all quality checks
npm run test:coverage                            # per-file line/branch/func report
NODE_ENV=test node --import tsx --test src/backend/auth-handler.test.ts  # single test file
```

## Critical Gotchas
- **Tests use Node.js built-in test runner** (`node --test`), not Jest/Vitest
- **`NODE_ENV=test` and config**: `src/config.ts` is a Proxy that throws on any property read in test mode. Tests must not transitively import `#root/config`. When a command needs `isAdmin` or `tonClient`, split the file into `foo-handler.ts` (pure, testable) + `foo.ts` (composer wiring that injects the heavy deps).
- **`folderPath()`** from `src/common/helpers/files.ts` for all file path ops — sanitizes usernames, checks against `./data/`
- **Claim locking**: in-process promise chain (`claimLocks` Map) — single-process only
- **Path aliases**: `#root/*` → `./build/src/*` (backend), `@/*` → `./src/*` (frontend)
- **ESM only**, no CommonJS. No semicolons, single quotes, 2-space indent.

See `auth-handler.test.ts` (route handler) or `help-handler.test.ts` (bot command) for reference.

For handlers whose default deps would transitively import `#root/config`, split into a pure handler module + a composer-wiring module that imports the heavy bits. Examples: `admin/user-handler.ts` + `admin/user.ts`, `admin/transaction-handler.ts` + `admin/transaction.ts`.

## Auth
All authenticated endpoints validate Telegram's `initData` string (HMAC + 24h expiry) → extract `user.id` → lookup in MongoDB.

### Wallet binding (TON Connect ton_proof)
`POST /api/auth/set-wallet` requires cryptographic proof the caller controls the wallet they want to bind. Flow:

1. Frontend calls `POST /api/auth/wallet-nonce` with `initData` and receives `{ payload, validUntil }`. The payload is a stateless HMAC token (`<userId>:<expiresAtMs>:<rand>:<hmac(BOT_TOKEN)>`) with a 5-minute TTL.
2. Frontend passes the payload to `TonConnectUI.setConnectRequestParameters({ state: 'ready', value: { tonProof: payload } })` before opening the wallet modal.
3. On connect, the wallet returns `account.{address,publicKey,walletStateInit}` and `connectItems.tonProof` (a `TonProofItemReplySuccess`).
4. Frontend POSTs to `/api/auth/set-wallet`: `{ initData, address, publicKey, walletStateInit, proof: { timestamp, domain, payload, signature } }`.
5. Backend (`src/backend/ton-proof.ts → verifyProof`) checks: payload HMAC + expiry + userId match, wallet timestamp within ±5 min, domain equals host of `WEB_APP_URL`, address hash equals `Cell.fromBoc(walletStateInit)[0].hash()`, Ed25519 signature valid over the canonical TON Connect message.

Each rebind requires a fresh nonce. The payload is bound to `userId`, so a stolen nonce can't be replayed against a different account.

## Bot middleware order
`autoRetry → updateLogger (dev only) → autoChatAction → hydrate → session → slapReaction → i18n → attachUser → queueMenu → [features]`

Active features (registered in `src/bot/index.ts`): start, help, queue (admin), parameters (admin), collection (admin), stats, whales, line, transaction (admin), user (admin), seasonPass (Telegram Stars `pre_checkout_query` + `successful_payment`). Trailing safety net: `removedCommandsFeature` catches any `/dice`, `/mint`, `/play`, etc. and points the user to the Mini App; `unhandledFeature` is the final catch-all.

## Game mechanics
- `addPoints()` uses `$inc` + logs to Balance model
- Daily claim: 60s cooldown, 10-day streak, 100 base × multiplier
- CNFT type at mint time (`pickCNFTType` in `src/common/models/CNFT.ts`): Whale (>1M votes), Diamond (>500K), Coin (>100K), Knight (>0 referrals), Common. `Dice` is a legacy type still in the enum and on past records but no longer produced — the `/dice` command was removed and `diceWinner` is never set to true for new users.
- `BalanceChangeType` enum (`src/common/models/Balance.ts`): Initial, Deposit, Withdraw, Dice, Referral, Donation, Task, Claim, Trade, Expedition, Spend. `Dice` and `Task` are legacy values; new entries are Claim/Referral/Donation/Trade. `Expedition` is the expedition-loop CUBE faucet (settlement payout); `Spend` is its CUBE sink (energy refill / weight-boost / tournament entry). USDT rewards-pool movements use the separate `RewardsEntryType` (not `BalanceChangeType`).
- Expedition economy core loop (`#root/common/helpers/{tick,energy,congestion}` + `{Energy,World,Expedition}` models + `/api/game/{worlds,expedition,energy/refill}` handlers): spend Energy to send an expedition to one of 5 cube-worlds per 1-hour tick; at tick close each world's fixed CUBE pool is split among its explorers by effective weight (crowd-dilution congestion), credited via the idempotent settlement runner (`src/backend/expedition-settlement.ts` pure + `expedition-settlement-runner.ts` composer, started in `main.ts`). The CUBE faucet credit is gated behind `EXPEDITION_FAUCET_ENABLED` (config, **default off**): until enabled, expeditions still resolve/display shares but mint no CUBE — flip on once the three sinks are tuned, and the uncredited backlog pays out. See `docs/ECONOMY.md` and the plan in `Journal/Projects/cube_worlds/plans/`.
- Monetization & payouts (Plan 3, `#root/common/helpers/{tournament,rewards,ad-nonce}` + `{Tournament,TournamentEntry,RewardsPoolLedger,SeasonPass,AdGrant}` models):
  - **Three CUBE sinks now exist** (refill, weight-boost, **tournament entry** `TOURNAMENT_ENTRY_CUBE`), satisfying the sinks-before-faucet invariant that gates the faucet above.
  - **Rewards pool** = 20% of net revenue (`REWARDS_POOL_BPS`), an append-only **`RewardsPoolLedger`** in **signed micro-USDT**, idempotent on `externalId` (`poolBalance()` = Σ rows). Accrued (best-effort, never fails the user action) on buy-energy, rewarded-ad views, and Season Pass charges. `RewardsEntryType` is its enum — distinct from `WalletEntryType`/`BalanceChangeType`.
  - **Weekly tournament** (`tournament.ts`, Monday-aligned 7-day `weekId`): entry is the CUBE sink (one entry/user/week, unique index); free for active Season Pass holders. `/api/game/{tournament,tournament/enter}` (pure `tournament-handler.ts` + composer `tournament.ts`). The settlement worker (`tournament-settlement.ts` pure + `tournament-settlement-runner.ts` composer, started in `main.ts` when the money rail is enabled) ranks entrants by in-window expedition CUBE (`Balance` aggregation), splits the pool by `PAYOUT_WEIGHTS_BPS`, and pays via free xRocket `transfer` — idempotent per `payout:<weekId>:<userId>` (`claimPayout` CAS + ledger `externalId`), paused while withdrawals are paused, gated by a 3-day account-age anti-Sybil check.
  - **Rewarded ads** (`/api/game/{ad-nonce,ad-reward}`, pure `ad-reward-handler.ts` + composer `ad-reward.ts`): `/ad-nonce` issues a single-use, 10-min HMAC nonce (`ad-nonce.ts`, secret = `ADSGRAM_REWARD_SECRET`); Adsgram's S2S `GET /ad-reward?payload=` verifies it, enforces `AD_DAILY_CAP`/UTC-day (`AdGrant`, unique on `nonceRand` = single-use), grants `AD_ENERGY_REWARD` energy, accrues ad revenue. The nonce — not any client-supplied id — is the integrity boundary.
  - **Season Pass** (`SeasonPass` model, idempotent on `telegram_payment_charge_id`): `POST /api/game/season-pass/invoice` returns a Stars invoice link (`createSeasonPassInvoiceHandler(bot.api)`, `provider_token=''`, `currency='XTR'`, `subscription_period=2592000`); the `seasonPass` bot feature grants/extends the pass on `successful_payment`. Active pass raises the energy cap to `SEASON_PASS_ENERGY_CAP` — `grantEnergy`/`spendEnergy`/`getEnergyStatus` resolve the per-user cap (`energyCapFor`) and never clamp a stockpile below the stored balance.

- Castle / Phase A (Ancient Worlds): `Castle` model (`src/common/models/Castle.ts`) — one per user (unique `userId`); 4 upgrade tracks `walls/forge/tavern/mine`; 4 DB-only resource balances `gold/iron/mana/food`; `lastProductionAt` clock; `nftMinted`/`nftAddress`. Helpers: `findOrCreateCastle`, `readBag/addBags/subtractBags/canAfford`, `creditProduction`, atomic `spendForUpgrade` ($gte CAS), `upgradeCost(track, level)` = base×(level+1), `MAX_TRACK_LEVEL=10`. Production tick (`src/common/helpers/production.ts`): `PRODUCTION_TICK_MS=8h`, `computeProduction` accrues lazily, caps at `MAX_UNCLAIMED_TICKS=3` (24h), Mine level scales output (1+0.25×level, floored per-tick); `BASE_PRODUCTION = {gold:50,iron:30,mana:10,food:40}`. Founder perk (`src/common/helpers/founder.ts`): `isFounder(user)` keyed on `User.minted` (existing CNFT holders) → `FOUNDER_MULTIPLIER=1.2` (+20% production). Resource audit trail: `ResourceLedger` model (`src/common/models/ResourceLedger.ts`) — append-only, `ResourceChangeType {Production,Upgrade,Loot}`. Endpoints (pure-handler + DI, prefix `/api/game`): `production-handler.ts` → `POST /castle` (status), `POST /castle/claim` (credits production + ledger); `castle-upgrade-handler.ts` → `POST /castle/upgrade` (`BalanceChangeType.CastleUpgrade` CUBE sink + atomic resource debit via $gte CAS, refund on CAS loss). Frontend: `CastleComponent.vue` is the home tab (🏰 at `/`); old daily-claim page moved to hidden `/claim`; expeditions relabelled "Dispatch" at `/expeditions`. Castle NFT mint: pure `castle-mint.ts` (`planCastleMints`, `buildCastleMintRunner`, mint-before-flip bias) + composer `castle-mint-runner.ts` + chain seam `castle-nft-client.ts`; started in `main.ts` every 5 min, gated by `CASTLE_COLLECTION_ADDRESS`. On-chain $CUBE bridge: pure math `cube-bridge.ts` (`WITHDRAW_FEE_BPS=200` = 2%, `WITHDRAW_COOLDOWN_MS=24h`, `canWithdraw`, `applyWithdrawFee`); idempotent `CubeBridgeLedger` (unique `externalId`, status pending/completed/failed, micro-CUBE as String); handler `cube-bridge-handler.ts` → `POST /api/cube/{status,withdraw}` (debit-first, refund + mark failed on chain-send throw); composer `cube-bridge.ts` + chain seam `cube-jetton-client.ts` (`isCubeJettonEnabled`, stub `sendCubeJetton`); route gated by `CUBE_JETTON_MASTER`+`CUBE_VAULT_ADDRESS`. DB ledger stays canonical; on-chain is for transfers only. **Operational note:** if chain-send succeeds but `markBridgeStatus(Completed)` throws, the ledger row stays `pending` while funds are delivered — bias is intentional (false-error ≫ false-success). **Monitor/alert on long-lived `pending` CubeBridgeLedger rows.**

- Heroes & PvE / Phase B (Ancient Worlds, core slice): `Hero` model (`src/common/models/Hero.ts`) — DB-canonical, `userId`/`heroClass`/`level`/`xp`, first hero `soulbound`, `founderVariant` for CNFT holders, deploy-gated `nftMinted`/`nftAddress`. Pure helpers: `hero.ts` (`HERO_CLASSES = knight/mage/archer/rogue`, `BASE_STATS`+`PER_LEVEL` per class, `MAX_HERO_LEVEL=30`, `statsForHero`, `xpToReach`/`levelFromXp`/`applyXp`, `recruitCost(n)` = `{cube:1000n×(n+1), gold:300×(n+1)}`, `tavernCapacity(lvl)=1+lvl`); `combat.ts` (mulberry32 `makeRng`/`hashSeed`, `resolveCombat`/`resolveHeroCombat` — deterministic, 1000×-reproducible, `MAX_ROUNDS`-capped); `dungeon.ts` (`dayBucket`, `dungeonSeed(userId,heroId,day)`, level-scaled `dungeonEnemy`, `dungeonLoot` resource-only, `DUNGEON_XP_WIN=60`/`DUNGEON_XP_LOSS=20`). **Tavern recruitment** (`hero-handler.ts` → `POST /api/game/{heroes,recruit}`): capacity-gated by Tavern level, atomic CUBE sink (`debitVotes`, `BalanceChangeType.Recruit`) **then** Gold sink (`spendResources` $gte CAS, `ResourceChangeType.Recruit`) with **refund-on-loss** (mirrors `castle-upgrade-handler`). **Daily dungeon** (`dungeon-handler.ts` → `POST /api/game/{dungeon,dungeon/run}`): resolves the deterministic seeded fight vs a level-scaled enemy; `DungeonRun` model unique `(userId, day)` claim + `credited` CAS flip = **exactly-once** loot (`creditResources` + `ResourceChangeType.Loot`) + XP (`applyXp`→`grantHeroXp`). **Loot is resources + XP only — NO new CUBE faucet** (CUBE loot deferred, would need `EXPEDITION_FAUCET_ENABLED`-style gating). Hero NFT mint: pure `hero-mint.ts` (`planHeroMints`, `buildHeroMintRunner`, re-entrancy guard + mint-before-flip bias) + composer `hero-mint-runner.ts` + chain seam `hero-nft-client.ts` (`isHeroMintEnabled`, stub `mintHeroNft`); started in `main.ts` every 5 min, gated by `HERO_COLLECTION_ADDRESS`. - Equipment / 8h Quest / Boss Week / Plan 6 (Ancient Worlds): three deferred Phase-B features, all DB-canonical with deploy-gated on-chain mint, **no new CUBE faucet** (loot is resources + XP + rate-limited equipment items). **Equipment** (`Equipment` model, pure `equipment.ts`): 4 slots `head/body/weapon/accessory` × 4 rarities `common/rare/epic/legendary`; `equipmentBonus(slot,rarity)` snapshotted onto the item at creation (a later table rebalance never restats owned gear); atomic `equipItem` CAS + **unique partial `(equippedHeroId, slot)` index** = one item per hero-slot; `equipment-handler.ts` → `POST /api/game/{equipment,equipment/equip,equipment/unequip}`. ## Money rail (xRocket)
Real-balance (USDT) layer, independent of the CUBE ledger. All USDT is stored as **bigint micro-USDT** (`1 USDT = 1_000_000n`) in the `WalletBalance` model — never a float. Conversions happen only at the xRocket boundary via `src/common/helpers/wallet.ts` (`usdtToMicro`/`microToUsdt`/`validateUsdtAmount`).
- **Ledger:** the append-only `WalletLedger` is idempotent on a unique `externalId` (xRocket payment id for deposits; a generated `randomId()` for debits) — a replayed webhook or retried payout collides on E11000 and is a no-op. `WalletEntryType` (deposit/withdraw/transfer/buy_energy) is the USDT enum; `BalanceChangeType.Deposit/Withdraw` remain **CUBE-ledger** values and are **not** used by the USDT rail.
- **Deposits** arrive via the signed `POST /api/wallet/webhook` (`HMAC_SHA256(sha256(BOT… xRocket token), rawBody)` in header `rocket-pay-signature`, verified timing-safe over the **raw** body by an encapsulated JSON parser scoped to that plugin only). Other endpoints: `/api/wallet/{balance,invoice,buy-energy,withdraw,transfer}`.
- **Overdraft is impossible:** debits go through an atomic `{ balance: $gte amount }` CAS (`applyDebit`). Withdraw/transfer refund the balance and mark the ledger row `failed` if the xRocket call throws.
- **Reconciliation:** the hourly worker (`wallet-reconciliation.ts` pure + `wallet-reconciliation-runner.ts` composer, started in `main.ts` when the rail is enabled) compares the completed-ledger sum against xRocket custody (`/app/info`) and flips the `WalletGuard` flag to **pause withdrawals/transfers** on divergence or missing custody balance.
- **Config seam:** `src/backend/xrocket-client.ts` is the only new module importing `#root/config` (`XROCKET_API_KEY` empty ⇒ rail disabled; `XROCKET_BASE_URL` defaults to testnet). Every wallet handler and the reconciliation worker are split pure-handler + composer so nothing a test imports touches config.
